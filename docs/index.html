8086tiny · Documentation
解説

8086tinyは小さくフリーでオープンソースのポータブルなIntel PCエミュレータ/VMであり、DOSやWindows 3.0, Excel, MS Flight Simulator, AutoCAD, Lotus 1-2-3などのアプリケーションを実行できるパワーを持っています。8086tinyは、次の機能を備えた「80年代後半の」PC XTタイプのマシンをエミュレートします。

  • Intel 8086/186 CPU
  • 1MB RAM
  • 3.5" フロッピーディスクコントローラ (1.44MB/720KB)
  • 固定ディスクコントローラ (最大520MBのハードディスクを1台サポート)
  • 720x348 2色および320x200 4色グラフィックス（64KBビデオRAM)、CGA 80 x 25 16色テキストモードをサポートするCGA/Herculesグラフィックカード
  • 正確なプログラマブル割り込みタイマー (PIT)
  • 83キーXTキーボードのキーボードコントローラstyle keyboard
  • リアルタイムクロック
  • PCスピーカー

このエミュレータはポータビリティを図るためにSDLグラフィックライブラリを使用し、様々なプラットフォーム（Windows, Mac OS X, Linux, Android, iOS, Raspberry Pi）でコンパイルできます。

提供している8086tinyは8086命令セットしか実装していませんが、より複雑で現代的な命令セットにも比較的容易に拡張できます。

ビルド方法

8086tinyディストリビューションには、ほとんどのUNIXプラットフォームで変更なしにコンパイルできるMakefileが含まれています。8086tinyのソースコードはMicrosoft Visual Studio C/C++でも変更なしにコンパイルできます。

  • Running makeを実行すると、8086tinyディストリビューション全体がコンパイルされます。これにはSDLによるオーディオとCGA/Herculesグラフィックスのサポートが含まれます。
  • （Raspberry Piなどの）低速なプラットフォームでは、グラフィックスエミュレーションフレームレートを上げるためにmake 8086tiny_slowcpuでビルドします。
  • 使用のプラットフォームにSDLがない、または、グラフィックモードやオーディオが必要ない場合は、make no_graphicsを実行することでグラフィックスやオーディオのサポートがない、よりコンパクトなバイナリを作成することができます。
使用方法

8086tiny bios-image-file floppy-image-file [@][harddisk-image-file]

harddisk-image-fileの頭に@を付けると8086tinyはハードディスクから起動します。それ以外はフロッピーディスクイメージから起動します。

UNIXでは、エミュレータを実行するためにsttyコマンドを使ってキーボードをrawモードする必要があります。ディストリビューションには、キーボードモードを適切に設定し、フロッピーディスクまたはハードディスクからエミュレータを実行するrunmeという名のスクリプトが含まれています。

#!/bin/sh
clear
stty cbreak raw -echo min 0
if [ -f hd.img ]
then
  ./8086tiny bios fd.img hd.img
else
  ./8086tiny bios fd.img
fi
stty cooked echo
ハードディスクイメージの作成

エミュレータで使用するハードディスクイメージを作成するには、mkfileなどのツールを使って、 必要なサイズのフラットなファイルをたとえばhd.imgという名前で作成し0バイトで埋めます。

DOSにおいてエミュレータで使用するハードディスクイメージを用意するには、実際のPCと同じように行います。

  • エミュレータを起動し、FDISKを使ってハードディスクのパーティションをします。 パーティションが終わるとFDISKはエミュレータを再起動します。
  • FORMAT C:（ブート可能なディスクを作成する場合はFORMAT C: /S）を実行して、 ディスクイメージをフォーマットします。これで終わりです。

出来上がったディスクイメージは、Windows PCではOSFMountで、Macではhdiutilで、 Linuxではmountでマウントできるフォーマットであり、ディスクにファイルやプログラムを簡単にコピーできます。 また、通常のフロッピーディスクイメージ（以下の「フロッピードライブエミュレーションを参照）を使って、エミュレータの中から プログラムをインストールできます。

キーボードサポート

このエミュレータは、I/Oポート0x60にあるキーが押される度に割り込み9を発生するIntel 8042チップにより制御されるXTスタイルのキーボードをシミュレートします。現実の8042はASCII文字ではなくスキャンコードを返すので、ポータビリティのために、エミュレータのBIOSは現実のPCの BIOSとは逆に、ASCII文字をスキャンコードに変換し、「現実の」キーボートと同じように機能するために必要に応じて修飾キー（シフトキーなど）の押す/放すをシミュレートします。その後、OS(DOS/Windows)はスキャンコードをASCII文字に戻すので、通常、このプロセスはシームレスに行われます。

BIOSにあるスキャンコード表は手元のキーボードレイアウトをUSレイアウトのQWERTYキーボードに合うよう対応付けます。国際化キーボードを レイアウトを使用している場合、使用しているOS（Windows 3.0など）が提供している「米国83キー・キーボード」などを選択することで、変更なしにすべてがうまく動くはずです。

コンソール（テキスト）アプリケーションで、Alt+xxxやFxx、Ctrl+xxxといった特別なキーシークエンスがあります。これらは標準CのI/O関数では直接返せないので、次のようにします。

  • Alt+xxxキーコンビネーションを送信するには、Ctrl+Aを押してから、該当のキーを押します。たとえば、Alt+Fを入力するには、まずCtrl+Aを押し、続いてFを押します。
  • Fxxキーを送信するには、Ctrl+Fを押してから、数字キーを押します。たとえば、F4を入力するには、まずCtrl+Fを押し、続いて4を押します。F10を入力するには、まずCtrl+Fを押し、続いて0を押します。
  • Ctrl+Aを送信するには、Ctrl+Fを押してから、Ctrl+Aを押します。Ctrl+Fを送信するには、Ctrl+Fを押してから、Ctrl+Fを押します。
  • Page Downキーを送信するには、Ctrl+Fを押してからOを押します。 Page Upキーを送信するには、Ctrl+Fを押してからQを押します。

グラフィック（SDL）アプリケーションでは、上のような特別なシークエンスなしに、すべてのキーが「現実の」PCどおりに機能します。

キーボードはKEYBOARD_TIMER_UPDATE_DELAY命令ごとにポーリングされます。この値を下げるとキーボードの反応は早くなりますが、エミュレートするCPU速度が犠牲になります。逆も同じです。デフォルトの20,000はおとんどのプラットフォームに適しているはずです。

フロッピードライブエミュレーション

3.5"高密度フロッピードライブをエミュレートします。1.44MBディスク（1トラック18セクタ、2ヘッド）と720KBディスク （1トラック9セクタ、2ヘッド）の読み込み、書き込み、フォーマットが可能です。

（Vetuswareなどからダウンロードした）複数枚数のフロッピーイメージからソフトウェアを インストールしたい場合に、「ディスク交換」を行う最も簡単な方法は、8086tinyを実行している端末とは違う端末から、コマンドラインで 指定したフロッピーイメージの順番とおりに各ディスクイメージをコピーすることです。最後にオリジナルブートディスクを指定することを 忘れないでください。

Hard Drive Emulation

最大1,023シリンダ、1トラック当たり63セクタ、63ヘッダで最大528MBのディスク容量をサポートします。

使用するディスクイメージフォーマットはほとんどのディスクイメージマウントツールで使用されている標準的な"raw" フォーマットのサブセットです。一般に、このエミュレータで用意されたディスクイメージはディスっくイメージツールや他のエミュレータで動作しますが、それ以外の方法では動作しません。

このエミュレータは、あまりにも単純なアルゴリズムを使って、ディスクイメージファイルのサイズからシミュレートするシリンダ/セクタ/ヘッダのジオメトリを抽出します。このアルゴリズムにより、通常、イメージファイル内のすべての領域がディスクパーティションとして使えるわけではありません。たとえば、40MBのイメージをファイルを作成しても、DOS FDISKで見るとボリュームサイズは31.98MBにしかなりません。

8086tinyはコマンドラインのハードディスク指定の際にファイル名の前に@を付けるとハードディスクイメージから起動します。たとえば、./8086tiny bios fd.img @hd.img

テキストモードのサポート

このエミュレータは、標準的なBIOS0x10割り込みインターフェイスを介したテキスト出力をサポートし、また、ダイレクトビデオメモリアクセス（1ページ、セグメントB800の4KBのビデオRAM）による80 x 25のCGA 16色テキストモードをサポートします。

BIOSのテキスト出力呼び出しは単純なstdoutへの単純なstdoutへの書き込みに変換されます。 ダイレクトビデオメモリアクセスによる80 x 25のCGAカラーテキストモードは、ANSI端末エスケープシーケンスに変換されます。 ANSIをサポートしていない端末（MS VC++でエミュレータをコンパイルしたり、Windowsのコンソールウィンドウで実行している場合など）を使用している場合は、テキストモードでビデオメモリに直接書き込むPCアプリケーションは使用できない可能性があります。端末ウィンドウのサイズは少なくとも80 x 25であることを確認してください。

テキストモードのビデオメモリは、8 * KEYBOARD_TIMER_UPDATE_DELAY命令ごとに端末にレンダリングされます。この値を小さくすると、テキストの更新速度は向上しますが、エミュレートされたCPU速度は犠牲になります。逆もまた真です。デフォルト値の20000は、ほとんどのプラットフォームに適しているはずです。

通常のPC文字コードページ（437）には、線画などのためのさまざまな拡張ASCII文字が含まれています。使用している端末プログラムにこれらを含むフォントに設定したいかもしれません（たとえば、Mac OS Xには、Perfect DOS VGA 437と呼ばれる最適なフリーフォントがあります）。そうしないと、拡張文字が正しく表示されない（たとえば、疑問符記号など）ことがあります。

時折、DOSアプリケーションがビデオハードウェアをおかしな状態にして終わることがあり、エミュレータを混乱させ、その結果、その後のテキスト出力が見えなくなることがあります。このような場合は、DOS　CLSコマンドを使用して画面をクリアしてください。すべてがうまくいきます。

グラフィックモードのサポート

Hercules 720x348モノクログラフィックスとCGA 320x200 4色グラフィックモードをSDLを使ってエミュレートします。ほとんどのCGA / Hercules機能を、0x3Dx と0x3Bxポート上の一般的なI/Oインターフェースを通じてサポートしています。 また、ダブルバッファグラフィックスを使用するユーザのためにビデオメモリバンクの切り替え（セグメントB000/B800）もサポートしてます。たとえば、640x408を使用するETEN中国語システムなどで必要なCRTCレジスタによる解像度再プログラミングは、8086tiny 1.03以降でサポートされています。現在、CGA 640x200の2色モードは現在サポートされていません。

アプリケーションがグラフィックモードに入ると、エミュレータはSDLウィンドを開きます（アプリケーションがテキストモードに戻ると閉じます）。UNIXでは、DISPLAY環境変数が設定されている場合、SDLは自動的にX11を通じてグラフィックスを出力します。

グラフィックス表示は、GRAPHICS_UPDATE_DELAY命令ごとに更新されます。この値を小さくすると、グラフィックスの更新速度は向上しますが、エミュレートするCPU速度は犠牲になります。逆もまた真です。GRAPHICS_UPDATE_DELAY命令のデフォルトは360,000に設定されており、高速なプラットフォームでは良好なパフォーマンスが得られます。 Raspberry Piのような低速なプラットフォームでは、より小さな値が適しています。make 8086tiny_slowcpuでビルドした場合、GRAPHICS_UPDATE_DELAYは50,000命令に減らされます。

（AutoCADなど）アプリケーションの中には、2つのディスプレイにテキストとグラフィックスを同時に出力するために、CGAカードとHerculesカードを両差したPC構成をサポートしています。このエミュレータは、（端末）テキストと（SDL）グラフィックス表示用に別々のウィンドウを使用することでこの構成もシミュレートします。

アプリケーションがテキストモードのみを必要とする場合は、 NO_GRAPHICSを定義することによりSDLなしに8086tinyをコンパイルできます。

リアルタイムクロックのサポート

RTC（時間と日付）の読み込みは、ホストコンピュータから時間/日付を取り込み、標準的なBIOSクロックインターフェイスを通じてエミュレートします。時間と日付の設定はサポートしていません。

ハードウェアタイマーのサポート

通常のI/Oポート0x40-0x43インターフェースを通じて、プログラマブル割り込みタイマーのチャネル0と2をエミュレートします。現在、サポートしているのはモード3（方形波ジェネレータ）だけですが、ほとんどのアプリケーションが使用しているおはこのモードです。タイマーを使用してゲームなどの実行速度を制御するソフトウェアは、正確なペースで実行する必要があります。

PCスピーカーのサポート

PCスピーカーは通常の0x61ポートインターフェースを通じてエミュレートしています。PCスピーカーでサポートされているモードはPITチャネル2経由のみです。そのため、ほとんどの音楽は聞くことができるでしょうが、非音楽のサウンド効果は聞けないでしょう。

BIOS

実際のPC同様に、エミュレータにはブート機能と標準的な割り込みインターフェイスを実装するためのBIOSが必要です。8086tiny BIOSは、パブリックドメインのドキュメントを使用して一から書きました。サイズは約6KBで、NASMでアセンブルしました。完全なソースコードとビルド済みのバイナリが用意されています。

BIOSバイナリは、コードセクションとデータセクションからなります。コードセクションでは、ビデオ、ディスク、タイマー、時計などの標準的な割り込みインターフェイスを「実際の」PC BIOSと同じように実装しています。また、エミュレータがテキストモードの場合、小さなタイマー制御のビデオドライバを使用してビデオメモリのフォーマットをANSIエスケープシーケンスに変換します。データセクションには、スキャンコードテーブルやBIOSデータエリアのような一般的なBIOS構造体だけでなく、エミュレータの命令解読を支援する多数のルックアップテーブルも含まれています。 以下の「CPU、メモリ、レジスタのエミュレーション」に詳細が記載されています。

メモリマップとレジスタのエミュレーション

エミュレータは、A20アドレス線のラップアラウンドを無効にしたハードウェア構成をシミュレートしており、アプリケーションは1MBをわずかに超えるRAMしか使用できません。

メモリマップは実際のPCとほとんど同じであり、割り込みベクタテーブルは0：0、BIOSデータ領域は、キーボードバッファが40:0、CGAテキストビデオメモリがB800:0、Hercules / CGAグラフィックメモリがB000/B800:0 、BIOSがF000:0100です。実際のPCとは異なり、エミュレータでは、CPUレジスタはメモリにマップされています（F000:0）。これにより、実行中の僧徒には知られることなく、メモリとレジスタ操作の統一ができ、エミュレータの命令実行ユニットを大幅に最適化することができます。

CPU、メモリ、レジスタのエミュレーション

CPUは、8086命令セットを完全にサポートしています。これには、文書化されていない命令（SALCなど）やフラグの動作（MUL/DIVなど）、一部のアプリケーションが依存するオペコードのバグ（PUSH SPなど）、デバッガをサポートするためのトラップフラグを含みます。

8086tinyの焦点は、エミュレーションの精度を損なうことなくコードサイズを最小限に抑えることです。きわめて不規則なIntel x86命令フォーマットの複雑さのために、命令のデコードはBIOSバイナリを形成する多くのルックアップテーブルによって支援されます。たとえば、ソースオペランドとデスティネーションオペランドのタイプ（即値、レジスタ、またはメモリ）に応じて、MOV命令をエンコードする方法はさまざまです。場合によっては、同じ命令をエンコードする複数の方法もあります（たとえば、MOV AX, [1234]は通常A1 34 12とエンコードしますが、8B 06 34 12とエンコードすることもできます）。各命令やエンコードバリアントに対してエミュレータが同じ機能を複数回実装しないようにするために、ルックアップテーブルを使用して、各命令を内部関数や副関数の番号にマッピングしています。

例として、16進数01 D8とエンコードされている命令ADD AX, BXをエミュレータがどのように実行するかを示します。

  • エミュレータはまず、（命令の最初のバイトである）インデックス01HをTABLE_XLAT_OPCODEとTABLE_XLAT_SUBFUNCTIONで検索し、変換済オペコードIDとして10進数の9（Intel命令テンプレートarithmetic_functionレジスタ、r/mに対応）と副関数IDとして0（ADD機能に対応）を得ます。
  • ソースのOPCODEチェーンが変換済オペコードIDと副関数IDを使用して実行するオペレーションを決定します。この場合、まずOP(+=)マクロを、次にset_CF()を呼び出し、加算の結果に基づきキャリーフラグを設定します。
  • 次に、命令長が計算されます。インテルx86命令は任意の長さであり（同じ命令の複数のエンコーディングが異なる長さを持つこともあります）、表を使用して命令長を決定し、IPを次の命令に移行します。オペコードインデックス01Hが、TABLE_BASE_INST_SIZEとTABLE_I_MOD_SIZE、TABLE_I_W_SIZEへのインデックスとして使用され、これらの数値が総命令長を計算するために加算されます。
  • 最後に、フラグが設定されます。オペコードインデックス01Hが、TABLE_STD_FLAGSへのインデックスとして使用され、ビットマスクが3になります。これはFLAGS_UPDATE_SZP | FLAGS_UPDATE_AO_ARITHを意味します。
    • FLAGS_UPDATE_SZP（1）は、この命令が演算の結果に従って標準的な方法で符号とゼロ、パリティの各フラグを設定することを示します。符号フラグとゼロフラグは結果から直接設定され、パリティフラグはTABLE_PARITY_FLAGの結果を参照して設定されます。
    • FLAGS_UPDATE_AO_ARITH（2）は、この命令が算術演算の基準として補助フラグとオーバーフローフラグを設定することを示します。
    • FLAGS_UPDATE_OC_LOGIC（4）がビットマスクに設定されている場合（この例では設定されていません）、ロジック操作の基準として、オーバーフローフラグとキャリーフラグが0に設定されます。

CPUは、エミュレータが外部世界と通信するのを補助するために次の"特別な"2バイトオペコードも実装しています。

  • 0F 00 (PUTCHAR_AL) - ALレジスタの文字を端末に出力する
  • 0F 01 (GET_RTC) - （localtimeが返す）リアルタイムクロックデータをES:BXが示すメモリ一に書き込む。
  • 0F 02 (READ_DISK) - ディスクのオフセット512*(16*SI+BP)からAXバイトをメモリロケーションES:BXに読み込む。ディスクはDLで指定される（0 = ハードディスク, 1 = フロッピーディスク）。
  • 0F 03 (WRITE_DISK) - メモリロケーションES:BXのAXバイトをディスクのオフセット512*(16*SI+BP)に書き込む。ディスクは0F 02と同じくDLで指定する。

エミュレータの終了はJMP 0:0命令によりトリガすることができ、ユーザが端末をシャットダウンすることなく容易にエミュレータを終了できます。

8086tinyがサポートしている命令セットの拡張は、上で説明したBIOSソース内のテーブルを適宜修正し、対応する新しいOPCODEブロックをCソースに追加することで実現できます。
